Mangrove Counting API

Submit a field walkthrough video, get back a deduplicated count of unique mangrove trees, an annotated video, and a top-down spatial map. Built for server-to-server integration.

Base URL https://cy6erlizard--treegens-motion-memory-web.modal.run

Overview

The API turns a single video of a planting site into an auditable tree count. It detects seedlings in every frame, tracks each one across frames, and counts unique trees rather than raw per-frame boxes, then corrects for the two ways a walkthrough double-counts: the operator walking a row and walking back, and zooming in and out.

It works in a single request: you POST a video and the same response carries the full result, including unique_count. No polling. Processing runs on GPU and takes roughly 30 to 90 seconds per clip, so the request stays open for that long; set your HTTP client's read timeout to a few minutes. Many videos from different sites are processed in parallel. For unusually long videos there is an optional async mode.

Authentication

The current deployment is open and requires no key, intended to be called from Treegens' backend over HTTPS. Do not expose the base URL directly to end users.

Adding a key. A shared Authorization: Bearer <token> header can be enabled on request, so only your backend can submit jobs. Ask and it will be turned on for your environment.

Quickstart

One call: POST the video, read the count from the response. The base URL is omitted below for brevity. --max-time 300 keeps the connection open while the video is processed.

Async mode (optional)

Add ?async=true to the submit call to get a job_id back immediately (202) instead of waiting. Use this only for unusually long videos that might exceed your request timeout. Then poll GET /jobs/{id} until the top-level status is done, and read the same result object.

Submit a video

POST/visits/{visit_id}/videos

Upload one video as multipart form data under the field name video. By default the request waits for processing and returns the full result in the response. The connection stays open for the processing time (about 30 to 90 seconds), so set a read timeout of a few minutes.

Path parameters

NameTypeDescription
visit_idstringYour identifier for the site visit. Echoed back on the response so you can correlate results.

Body

FieldTypeDescription
videofileThe walkthrough clip (mp4/mov). Required.

Query parameters

NameTypeDescription
asyncboolOptional, default false. When true, return a job_id to poll (202) instead of waiting. See async mode.

Response 200 OK (default, single request)

The result object is the same one described under The result object. With ?async=true this endpoint instead returns 202 with { job_id, status: "queued", poll }.

Get job status and result

GET/jobs/{job_id}

Used with async mode to poll a job, and to fetch a past result by id. While queued or running the payload is small; when done it carries the full result object.

Response 200 OK (done)

Get the annotated video

GET/jobs/{job_id}/video

Returns the browser-playable H.264 video/mp4: every tree boxed with its stable ID and a running unique count. Available once the job is done.

Get the spatial map

GET/jobs/{job_id}/path_image

Returns a image/png top-down map: the camera's reconstructed path coloured by what it was doing, the inferred tree positions, and any revisits that were merged. Present only when a path could be reconstructed (see result.path_image).

Health check

GET/health

The result object

Present on a done job under result. The fields your integration will use most are unique_count and requires_review.

FieldTypeDescription
unique_countintHeadline number: unique mangrove trees after all deduplication and correction.
requires_reviewboolTrue when the count is uncertain (no detections, heavy gap-filling, or ambiguous camera motion). Route these to a human verifier.
count_before_correctionintUnique track IDs before revisit and zoom correction. Always ≥ unique_count; the gap is what was merged.
confidencefloat0 to 1. Share of counted boxes that were real detections rather than interpolated fills.
naive_per_frame_sumintWhat a naive per-frame box count would report. Shown for contrast; hugely inflated, never a real count.
framesintFrames processed from the clip.
count_correctionobjectWhat the dedup stages merged: parked_zoom, respawn, spatial (revisit), and merged_total.
camera_motionobjectAdvisory summary of what the camera did: dominant_state, per-state fractions, uncertain_fraction, zoom-segment count.
camera_pathobjectReconstructed top-down path, inferred tree map, and n_revisits. Null when no path could be built.
gpsobjectlat, lon, accuracy. Videos carry no embedded GPS, so this is camera/site-level only, never per-tree.
metadataobjectContainer metadata (width, height, frame rate, duration) from ffprobe.
annotated_videostringRelative URL of the annotated count video.
path_imagestringRelative URL of the spatial map PNG, when produced.
model_versionstringModel that produced the count. Stamp it on stored results.
dataset_versionstringTraining dataset version behind the model.
timings_sobjectPer-stage wall-clock timings in seconds.
Count once, not per frame. unique_count is the only number to store as the tree total. naive_per_frame_sum is deliberately included so you can see the inflation the pipeline removes.

Errors

CodeWhen
200Single request finished; body carries the result.
202Async mode (?async=true): job accepted, poll for it.
400Missing or empty video field on submit.
404Unknown job_id, or an artifact requested before the job is done.

If processing itself fails, the response body carries status: "error" and a human-readable error message. Always branch on the top-level status field rather than the HTTP code alone.
